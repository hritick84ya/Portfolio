Design Principles in Practice

ARCHITECTURE & EXPERTISE

🧱

Trigger Framework & Org Architecture

  • One trigger per object — evaluated ff lib vs custom dispatcher, chose custom for lower dependency overhead
  • Service layer: all business logic lives here, callable from trigger, API, Flow, batch — single entry point
  • Selector layer: all SOQL centralised, reused across consumers, CRUD/FLS enforced at query construction
  • Context routing: before/after x insert/update/delete handled discretely — no conditional soup in handlers
  • Custom exception hierarchy: typed exceptions give ops teams actionable error categories, not stack traces
📐

Domain Layer & Design Patterns

  • Interfaces over concrete classes — every implementation is swappable, every dependency is mockable
  • Factory pattern: runtime rule selection without conditional chains that grow with every new requirement
  • Strategy pattern: business rule variants driven by Custom Metadata — new variant = new record, not a code change
  • Constructor injection: dependencies declared, not instantiated — classes are fully testable in isolation
  • Domain class per SObject: behaviour collocated with data, not scattered across triggers and utilities
⚡

Async Processing Strategy

  • Batch vs Queueable vs Future: decision based on data volume, callout need, chaining requirement, and mixed-DML risk — not convenience
  • Batch scope sizing: calibrated per-object against heap profile and CPU budget, not left at default 200
  • Queueable chains: every link is idempotent with a correlation ID — safe to re-execute on failure without duplicate processing
  • Scheduled Apex: health-check logging on every run — silent failures are caught before they cause data drift
  • Platform Cache: reference data (permission sets, config records) cached at org partition — repeat SOQL on hot paths eliminated
🔗

Integration Fault Tolerance

  • Sync vs async decision: synchronous REST only where immediate response is required by UX — everything else is event-driven or queued
  • Exponential backoff with jitter: prevents thundering-herd retry storms on external rate limits
  • Platform Events: producer doesn't wait for consumer — decoupling eliminates cascading failures across integration boundaries
  • Dead-letter queue: failed events persisted for replay, not silently dropped — every failure is visible and recoverable
  • Named Credentials + Connected Apps: auth configuration externalised — rotate credentials without a code deployment
🔒

Security Governance Architecture

  • Access modelling before configuration: OWD, Role Hierarchy, and sharing strategy designed on paper, reviewed with stakeholders, then implemented — not discovered post-deployment
  • CRUD + FLS enforcement: applied in Selector layer so every consumer of the query inherits the check automatically
  • with sharing by default, without sharing declared explicitly only where a business justification exists and is documented
  • Permission Set Groups model: access bundles are auditable, revocable atomically, and don't multiply Profiles with every role variant
  • Integration identity: dedicated Connected App, minimum-privilege profile, Named Credential — zero shared user accounts, zero hardcoded secrets
🧪

Test Strategy & Quality Architecture

  • Test design mirrors production architecture: unit tests hit service layer directly, integration tests exercise trigger → handler → service → selector as a unit
  • TestDataFactory: fluent builder pattern, no seeAllData=true, no org-specific IDs hardcoded — deployable to any sandbox without modification
  • Stub API + HttpCalloutMock: HTTP boundaries fully isolated — tests never depend on external service availability or response speed
  • Boundary testing: bulk boundaries (200+ records), governor-limit edge cases, and exception paths are first-class test scenarios
  • @TestSetup: single shared setup reduces test execution time 3–5× on large orgs — prevents database thrash across test methods
Architectural Philosophy

Design First.
Then Code.

Every architecture decision is a trade-off. I make them deliberately — async vs sync, event-driven vs polling, declarative vs programmatic, tight vs loose coupling. The right answer changes with context. What doesn't change: the decision gets documented and the org gets something maintainable.

Design Thinking in Practice

HOW I THINK & BUILD

🧱

Org Architecture First

Before any code, I design the org's structural layer — trigger dispatch, service boundaries, and data access patterns.

  • Trigger as router only — logic never lives here, so behaviour is testable independently
  • Service layer as the single entry point — called from trigger, Apex REST, Flow, Batch without duplication
  • Selector layer enforces SOQL hygiene and CRUD/FLS centrally — every consumer inherits the governance
  • Exception hierarchy designed to give ops teams actionable failure categories
📐

Extensibility by Design

I write code that a new developer can extend the day the original author leaves.

  • Interfaces over concrete classes — adding a new behaviour means implementing an interface, not modifying existing code
  • Factory + Strategy: new business rule variants via Custom Metadata, not code deployments
  • Constructor injection: no hidden dependencies, full testability in isolation
  • Domain classes: SObject behaviour collocated with the object — discoverable, not scattered
🧩

LWC Communication Decisions

I choose the communication pattern based on component topology — never by habit.

  • @api + CustomEvents: directional, explicit, low coupling — default for related component trees
  • LMS: selected when components have no shared ancestor — not used as a shortcut to avoid hierarchy design
  • @wire vs imperative: @wire for reactive data that should refresh on state change; imperative for conditional or user-triggered fetches only
  • State ownership mapped before markup is written — who owns state, who consumes it, what triggers refresh
🔒

Security as Architecture

I model the access architecture before writing any configuration — not after the first audit finding.

  • Access model on paper first: OWD, hierarchy, sharing strategy — agreed with stakeholders before configuration begins
  • CRUD + FLS in the Selector layer: every query consumer inherits enforcement automatically
  • Permission Set Groups: atomic, auditable access bundles — revoke cleanly without touching Profiles or other users
  • Integration identity: dedicated Connected App, minimum privilege, Named Credential — zero shared accounts, zero hardcoded secrets
🧪

Test Strategy as Quality Signal

I design the test architecture the same way I design the production code — layered, isolated, and meaningful.

  • Unit tests at service layer: fast, isolated, prove business logic without trigger overhead
  • Integration tests at trigger boundary: verify the full handler → service → selector pipeline as a unit
  • TestDataFactory with fluent builders: org-agnostic, no hardcoded IDs, deployable to any sandbox
  • Boundary tests are first-class: bulk (200+), governor edges, and exception paths are mandatory — not optional
🔗

Integration Architecture Decisions

I choose the integration pattern based on coupling risk, failure blast radius, and operational observability.

  • Sync vs async: synchronous only where UX demands immediate response — otherwise Platform Events or Queueable to decouple failure modes
  • Retry strategy: exponential backoff with jitter and a ceiling, configured via Custom Metadata — no hardcoded retry counts
  • Dead-letter queue: failed events persisted for replay — failures are visible and recoverable, never silently dropped
  • Every integration layer has structured error logging, a correlation ID, and an ops alert threshold — silent failures are not acceptable
